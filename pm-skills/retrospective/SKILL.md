# Retrospective - 项目复盘

---

## Description

结构化复盘项目/迭代，沉淀经验、发现问题、持续改进。

**解决的实际问题：**
- 复盘变甩锅大会，没有结论
- 问题发现了但不改，下次还犯
- 好的经验没有沉淀，新人重来

---

## When to Use

**强制触发条件（任一满足）：**
- 明确说"复盘"、"总结"、"回顾"
- 项目/迭代刚结束（1周内）
- 出现重大事故/延期
- 项目取得超预期成功

**不适用场景：**
- 项目进行中期（用进展同步代替）
- 纯个人工作复盘（范围太小）

---

## Pre-conditions

**必须提供：**
- 项目/迭代的基本信息（目标、时间、参与者）
- 实际结果（是否达成目标）

**可选提供：**
- 数据指标（预期 vs 实际）
- 团队反馈
- 用户反馈

---

## Workflow

### Phase 1: 事实回顾

**客观数据：**
```markdown
## 项目概况
- 项目名称：会员系统上线
- 周期：2026.03.01 - 2026.03.15
- 团队：产品1人 + 开发2人 + 测试1人

## 目标 vs 结果
| 指标 | 目标 | 实际 | 达成率 |
|------|------|------|--------|
| 上线时间 | 3月15日 | 3月18日 | ❌ 延期3天 |
| 注册转化率 | 20% | 15% | ❌ 未达标 |
| Bug数 | <5个 | 8个 | ❌ 超标 |
| 用户满意度 | - | 4.2/5 | ✅ 超预期 |
```

### Phase 2: 归因分析（KPT法）

```markdown
## Keep（继续保持）
1. 每日站会同步进度，问题早发现
   - 证据：第3天发现接口延迟问题，及时调整方案
   
2. 测试提前介入，编写用例
   - 证据：发现边界情况3处，上线后无P0 Bug

## Problem（存在问题）
1. 需求变更频繁
   - 现象：迭代中变更5次，导致开发返工
   - 根因：前期需求评审不充分，遗漏场景
   
2. 性能测试缺失
   - 现象：上线后高峰期响应慢
   - 根因：未做压测，高估了系统承载

## Try（改进尝试）
1. 需求冻结机制
   - 方案：迭代启动后，变更需走审批
   - 负责人：产品经理
   - 验证：下个迭代观察变更次数

2. 性能测试纳入流程
   - 方案：上线前必须压测，指标不达标不上线
   - 负责人：测试负责人
   - 验证：建立性能基线文档
```

### Phase 3: 行动项制定

**可执行标准：**
- [ ] 有明确的负责人
- [ ] 有完成时间
- [ ] 有验收标准
- [ ] 与问题根因对应

```markdown
## 行动项清单

| 问题 | 改进措施 | 负责人 | 完成时间 | 验收标准 |
|------|---------|--------|---------|---------|
| 需求变更频繁 | 建立需求冻结机制 | PM | 4月1日 | 迭代变更<2次 |
| 性能问题 | 压测纳入上线 checklist | 测试 | 4月1日 | 压测报告模板 |
| 沟通不足 | 每日站会15分钟 | 全员 | 立即 | 坚持1个月 |
```

### Phase 4: 经验沉淀

```markdown
## 可复用经验
1. **技术方案评审不可省**
   - 场景：涉及接口改造时
   - 做法：必须拉上后端和运维评审
   
2. **灰度发布兜底**
   - 场景：新功能上线
   - 做法：先5%灰度，观察24小时再全量

## 踩坑记录
1. **不要低估数据迁移成本**
   - 场景：历史数据兼容
   - 教训：预留50% buffer时间
```

## Output

```markdown
# 项目复盘报告：会员系统迭代

## 执行摘要
- 延期3天上线，核心功能可用
- 转化未达标，需优化引导流程
- 沉淀3条可复用经验

## 详细分析
[Keep/Problem/Try]

## 行动项
[清单]

## 经验沉淀
[可复用 + 踩坑]

## 下次迭代注意
1. 需求评审增加技术预评估环节
2. 预留性能优化时间
3. 转化指标拆解到每日监控
```

---

## Quality Gates

- [ ] 行动项 < 5个（聚焦关键）
- [ ] 每个问题都有根因分析（不只是现象）
- [ ] 有具体数据支撑（非主观感受）
- [ ] 行动项可验收（非"加强沟通"等空话）
