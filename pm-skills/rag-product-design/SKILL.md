# RAG Product Design - RAG 产品设计

---

## Description

设计基于 RAG（检索增强生成）的 AI 功能，让 AI 基于私有知识回答问题。

**解决的实际问题：**
- 通用 AI 不知道企业私有知识
- AI 幻觉严重，答案不可靠
- 需要基于最新/内部数据回答

---

## When to Use

**强制触发条件（任一满足）：**
- 明确说"RAG"、"知识库问答"
- AI 需要基于私有文档回答
- 需要减少 AI 幻觉
- 需要引用来源

**不适用场景：**
- 纯通用知识问答（无需 RAG）
- 数据量极小（<100 篇文档）
- 实时性要求极高（RAG 有延迟）

---

## Pre-conditions

**必须提供：**
- 知识库内容（文档类型/数量）
- 用户问题类型
- 准确性要求

**可选提供：**
- 现有搜索/知识库系统
- 数据更新频率

## Workflow

### Phase 1: 场景与数据评估

```markdown
## RAG 适用性评估

### 数据条件
| 条件 | 要求 | 当前状态 |
|------|------|---------|
| 文档数量 | >100 篇 | ? |
| 文档质量 | 结构化/半结构化 | ? |
| 更新频率 | 可接受延迟 | ? |
| 数据权限 | 可访问处理 | ? |

### 问题类型
适合 RAG：
- ✅ 事实性问题（"公司的年假政策是什么"）
- ✅ 文档摘要（"总结一下这份报告"）
- ✅ 多文档关联（"对比 A 和 B 方案"）

不适合 RAG：
- ❌ 创意生成（"写一首诗"）
- ❌ 推理计算（"帮我算个税"）
- ❌ 主观建议（"我该选哪个"）

### 本次评估
场景：企业内部知识库问答
数据：5000+ 文档，PDF/Word/网页
问题：政策查询、流程咨询、技术文档
结论：✅ 高度适合 RAG
```

### Phase 2: 架构设计

```markdown
## RAG 架构

### 数据流
```
用户提问
    ↓
Query 改写（优化检索）
    ↓
向量检索（Top-K 相关文档）
    ↓
重排序（精排）
    ↓
上下文组装（Prompt 构建）
    ↓
LLM 生成答案
    ↓
后处理（添加引用）
    ↓
返回答案
```

### 关键模块
| 模块 | 功能 | 选型 |
|------|------|------|
| 文档处理 | 解析/切分/向量化 | LangChain |
| 向量数据库 | 存储/检索向量 | Pinecone/ Milvus |
| 嵌入模型 | 文本转向量 | text-embedding-3 |
| LLM | 生成答案 | GPT-4 / Claude |
| 重排序 | 精排检索结果 | Cross-encoder |

### 优化策略
- 混合检索：向量 + 关键词
- 查询改写：扩展同义词
- 重排序：Cross-encoder 精排 Top-20
- 上下文压缩：只保留最相关段落
```

### Phase 3: 体验设计

```markdown
## 问答体验

### 1. 引用设计
```
答案：员工年假为 15 天/年 [1]

引用：
[1] 《员工手册》第 3 章第 2 节 - 年假政策
```

### 2. 不确定处理
```
基于现有资料，没有找到相关信息。
建议：
1. 换个关键词试试
2. 咨询 HR 部门
3. 查看员工手册原文
```

### 3. 多轮对话
```
用户：年假多少天？
AI：15 天 [1]
用户：包括试用期吗？
AI：不包括，转正后才有 [2]
（上下文理解，关联之前的问题）
```

### 4. 反馈机制
- 👍/👎 答案质量
- "这个答案有用吗"
- 错误报告入口
```

### Phase 4: 效果评估

```markdown
## 评估指标

### 检索质量
| 指标 | 定义 | 目标 |
|------|------|------|
| 命中率 | 相关文档在 Top-K 中 | >90% |
| MRR | 第一个相关文档排名 | >0.7 |
| NDCG | 排序质量 | >0.8 |

### 生成质量
| 指标 | 定义 | 目标 |
|------|------|------|
| 准确率 | 答案事实正确 | >85% |
| 覆盖率 | 问题被回答比例 | >80% |
| 幻觉率 | 编造信息比例 | <5% |

### 用户满意度
- 满意度评分 > 4/5
- 重复提问率 < 20%
- 人工介入率 < 10%
```
