# PRD Writer - PRD文档撰写

---

## Description

将模糊的产品需求转化为结构清晰、可落地的 PRD 文档。

**解决的实际问题：**
- 需求描述碎片化，开发无法理解
- 遗漏边界情况，导致上线后问题频发
- 多方对齐成本高，反复修改

---

## When to Use

**强制触发条件（必须同时满足）：**
1. 用户明确说"写PRD"、"出需求文档"、"产品文档"
2. 已提供：产品背景 / 目标用户 / 核心功能 中的至少两项

**适用场景：**
- 新产品/功能立项
- 需求评审前文档化
- 跨团队对齐（产品→开发→测试）

**不适用场景：**
- 纯技术实现方案（用技术方案 skill）
- 仅一句话需求且无补充（先用需求澄清 skill）
- 运营/市场文案（用文案 skill）

---

## Pre-conditions

**必须提供的信息：**
```yaml
背景: 为什么要做这个？业务痛点是什么？
目标用户: 给谁用？用户画像或角色
核心功能: 要解决什么问题？
```

**可选补充：**
```yaml
竞品参考: 对标哪些产品？
数据指标: 预期提升什么指标？
时间约束: 上线 deadline？
```

## Workflow

### Phase 1: 需求完整性检查

**检查清单（必须全部通过）：**
- [ ] 背景描述 > 50 字（排除"优化体验"等空话）
- [ ] 目标用户明确（不是"所有用户"）
- [ ] 核心功能可验证（有完成标准）
- [ ] 成功指标可量化（或明确"暂不量化"）

**未通过处理：**
```
⚠️ 信息不足，无法撰写高质量 PRD

缺失项：背景、目标用户

请补充：
1. 这个需求要解决什么业务问题？（具体场景）
2. 主要给哪类用户使用？（角色/场景）
```

### Phase 2: 用户故事地图

```
作为 [用户角色]
当 [场景/触发条件]
我想要 [动作/能力]
从而 [获得的价值]

示例：
作为 运营人员
当 需要批量上传商品时
我想要 通过 Excel 导入并自动校验
从而 节省 80% 的人工录入时间
```

**检查规则：**
- 每个核心功能必须对应至少 1 个用户故事
- 用户故事必须包含"从而"（价值点）
- 避免技术实现细节（不说"调用API"）

### Phase 3: 功能规格书

**每个功能的规格结构：**
```markdown
## 功能名：批量导入

### 用户界面
- 入口位置：商品管理 > 批量操作 > 导入
- 页面布局：[描述或附图]
- 关键交互：上传 → 预览 → 确认 → 结果

### 业务规则
| 规则项 | 规则内容 | 异常处理 |
|--------|---------|---------|
| 文件格式 | 仅支持 .xlsx, 最大 10MB | 超限提示："文件过大，请分批上传" |
| 必填字段 | 商品名、价格、库存 | 缺失提示："第N行缺少商品名" |
| 价格范围 | 0.01 - 999999 | 越界提示："价格超出有效范围" |

### 边界情况
1. 空文件上传 → 提示"文件为空"
2. 全部行校验失败 → 显示错误报告，不提供确认按钮
3. 部分成功 → 显示成功/失败明细，支持导出错误行

### 数据埋点
| 事件名 | 触发时机 | 属性 |
|--------|---------|------|
| import_start | 点击上传 | file_size, row_count |
| import_success | 导入完成 | success_count, fail_count, duration |
| import_fail | 校验失败 | error_type, error_rows |
```

### Phase 4: 非功能需求

**必须考虑：**
```markdown
## 性能要求
- 导入 1000 行：响应 < 3s
- 导入 10000 行：响应 < 30s，支持异步

## 兼容性
- 浏览器：Chrome 90+, Edge 90+
- 移动端：不适用（仅 PC 端）

## 安全要求
- 文件类型白名单校验（不仅检查后缀）
- 敏感字段脱敏（手机号中间四位隐藏）

## 降级方案
- 大文件自动转异步，通过邮件通知结果
- 模板下载失败时，提供手动复制示例
```

### Phase 5: 版本规划

```markdown
## MVP（2周）
- 基础导入功能
- 仅支持标准模板
- 同步处理，上限 1000 行

## V1.1（+1周）
- 异步处理支持
- 错误报告导出
- 上限提升至 10000 行

## V1.2（后续迭代）
- 自定义字段映射
- 历史记录查询
```

## Output

**标准 PRD 结构：**
```markdown
# 需求名称

## 1. 背景与目标
（Why）

## 2. 用户故事
（Who & What & Value）

## 3. 功能规格
（How - 详细设计）

## 4. 非功能需求
（性能/安全/兼容）

## 5. 数据埋点
（How to measure）

## 6. 版本规划
（When）

## 7. 附录
- 原型链接
- 竞品截图
- 数据报表
```

---

## Quality Gates

**输出质量检查（必须全部通过）：**
- [ ] 开发看完后能开始技术方案设计
- [ ] 测试看完后能写测试用例
- [ ] 未解决的疑问 < 3 个
- [ ] 用户故事覆盖所有功能点
- [ ] 边界情况 ≥ 3 个

---

## Known Limitations

1. 无法替代用户调研（需用户提供真实用户反馈）
2. 复杂算法逻辑需技术团队补充
3. 视觉细节需设计师介入
