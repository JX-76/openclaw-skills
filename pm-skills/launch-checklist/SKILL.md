# Launch Checklist - 上线检查清单

---

## Description

系统化检查上线前准备，防止遗漏导致线上事故。

**解决的实际问题：**
- 上线后才发现配置错了
- 忘记通知相关方
- 监控没配，出问题不知道

---

## When to Use

**强制触发条件（必须满足）：**
- 功能/项目准备上线（发布前1-3天）

**不适用场景：**
- 纯技术重构（无业务影响）
- hotfix紧急修复（用hotfix流程）

---

## Pre-conditions

**必须提供：**
- 上线功能清单
- 上线时间
- 影响范围（用户/系统）

---

## Workflow

### Phase 1: 功能检查

```markdown
## 功能验证
- [ ] 核心流程测试通过（P0用例100%通过）
- [ ] 边界情况处理（空状态/超限/异常）
- [ ] 兼容性测试（浏览器/APP版本）
- [ ] 回归测试（原有功能未受影响）

## 配置检查
- [ ] 生产环境配置正确（数据库/缓存/第三方）
- [ ] 功能开关配置（灰度比例/白名单）
- [ ] 内容配置（文案/图片/价格）
```

### Phase 2: 技术准备

```markdown
## 性能与容量
- [ ] 压测通过（达到预期QPS）
- [ ] 容量评估（数据库/存储增长预估）
- [ ] 降级方案（异常时如何兜底）

## 监控告警
- [ ] 核心指标监控（转化率/错误率/响应时间）
- [ ] 告警阈值配置（P0功能异常立即告警）
- [ ] 告警接收人确认（值班表更新）

## 日志与追踪
- [ ] 关键节点日志（方便排查问题）
- [ ] 错误日志规范（包含上下文信息）
- [ ] 链路追踪（分布式调用链）
```

### Phase 3: 数据准备

```markdown
## 数据迁移
- [ ] 迁移脚本测试（回滚脚本也准备好）
- [ ] 数据验证（迁移后数据一致性）
- [ ] 备份（上线前全量备份）

## 数据埋点
- [ ] 核心指标埋点（验证上线效果）
- [ ] 埋点验证（事件正确触发）
- [ ] 报表配置（实时数据看板）
```

### Phase 4: 协同与沟通

```markdown
## 内部通知
- [ ] 客服培训（新功能FAQ）
- [ ] 运营同步（上线时间和节奏）
- [ ] 数据团队（埋点确认）

## 外部准备
- [ ] 用户通知（如需）
- [ ] 应用商店更新（APP版本）
- [ ] 第三方同步（支付/物流等接口方）

## 应急预案
- [ ] 回滚方案（5分钟内可回滚）
- [ ] 故障联系人（24小时在线）
- [ ] 上报机制（P0故障立即上报）
```

### Phase 5: 灰度策略

```markdown
## 灰度计划
- [ ] 灰度比例：5% → 20% → 50% → 100%
- [ ] 灰度时长：每阶段观察24小时
- [ ] 灰度用户：内部员工 → 种子用户 → 全量

## 观察指标
- [ ] 核心指标无异常波动（±5%以内）
- [ ] 错误率 < 0.1%
- [ ] 用户反馈无集中投诉

## 全量标准
- [ ] 灰度用户转化率符合预期
- [ ] 无P1级以上Bug
- [ ] 持续24小时稳定
```
