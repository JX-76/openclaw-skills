# Data Analysis - 数据分析

---

## Description

基于数据进行产品决策，用数据说话而不是凭感觉。

**解决的实际问题：**
- 有数据但不知道怎么分析
- 分析了很多但没结论
- 数据和产品决策脱节

---

## When to Use

**强制触发条件（任一满足）：**
- 明确说"分析一下"、"看看数据"
- 产品决策需要数据支撑
- 出现异常波动（留存/转化下降）
- 周报/月报需要数据洞察

**不适用场景：**
- 产品早期无数据（<100用户）
- 纯探索性需求验证（用访谈）

---

## Pre-conditions

**必须提供：**
- 分析目标（解决什么问题）
- 数据范围（时间/用户群）
- 核心指标定义

**可选提供：**
- 历史基线数据
- 竞品对比数据

---

## Workflow

### Phase 1: 定义分析目标

```markdown
## 分析目标模板

### 问题类型
| 类型 | 示例 | 分析方法 |
|------|------|---------|
| 描述性 | 上周新增多少用户？ | 统计汇总 |
| 诊断性 | 为什么留存下降了？ | 下钻分析 |
| 预测性 | 下月DAU会多少？ | 趋势外推 |
| 指导性 | 该优化哪个功能？ | 相关性分析 |

### 本次分析
- 类型：诊断性
- 问题：次日留存从40%降到35%
- 时间范围：过去7天
- 期望产出：下降原因+改进建议
```

### Phase 2: 数据收集与清洗

```markdown
## 数据检查清单

### 完整性
- [ ] 时间范围正确（无缺失日期）
- [ ] 用户群完整（无抽样偏差）
- [ ] 字段无大量缺失值

### 准确性
- [ ] 指标口径一致（和之前定义相同）
- [ ] 无异常值（如留存>100%）
- [ ] 和数据源核对抽样

### 清洗记录
| 问题 | 处理方式 | 影响样本数 |
|------|---------|-----------|
| 测试账号 | 排除user_id<1000 | -50 |
| 缺失值 | 用前值填充 | 0 |
```

### Phase 3: 分析方法选择

```markdown
## 常用分析方法

### 1. 对比分析
```sql
-- 留存对比：本周 vs 上周
SELECT 
    week,
    COUNT(DISTINCT user_id) as new_users,
    COUNT(DISTINCT CASE WHEN day2_active=1 THEN user_id END) * 1.0 / COUNT(DISTINCT user_id) as d2_retention
FROM user_retention
GROUP BY week
```

### 2. 漏斗分析
见 funnel-analysis skill

### 3.  cohort 分析
见 cohort-analysis skill

### 4. 相关性分析
```python
import pandas as pd

# 功能使用 vs 留存相关性
corr = df[['feature_usage', 'retention_day7']].corr()
# feature_usage 和 retention 相关系数 0.65 → 强相关

### 5. 细分分析
```markdown
按维度拆分：
- 渠道：自然 vs 付费
- 版本：v1.0 vs v1.1

- 设备：iOS vs Android
```

### Phase 4: 洞察提炼

```markdown
## 分析结论模板

### 发现
1. 次日留存下降5pp（40%→35%）
2. 主要下降来自Android端（-8pp）
3. iOS端无变化

### 根因
- Android v1.1版本发布后出现
- 崩溃率从0.5%上升到2%
- 主要集中在小米机型

### 建议
1. 立即修复小米机型崩溃（P0）
2. 增加Android兼容性测试
3. 建立机型分级测试标准

### 预期效果
- 修复后留存恢复至38-40%
```
