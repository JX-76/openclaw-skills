# Metrics Design - 指标设计

---

## Description

为产品/功能设计可衡量的指标体系，避免"感觉好用"但无法量化。

**解决的实际问题：**
- 老板说"提升用户体验"，但不知道用什么衡量
- 指标太多，不知道看哪个
- 指标设计不合理，导致行为扭曲

---

## When to Use

**强制触发条件（任一满足）：**
- 用户明确说"指标"、"度量"、"数据"
- 提到"怎么衡量成功"
- 需求中包含"提升"、"优化"等模糊目标

---

## Pre-conditions

**必须提供：**
- 产品/功能目标

**可选提供：**
- 现有数据情况
- 行业参考

---

## Workflow

### Phase 1: 目标拆解

**将业务目标转化为可衡量指标：**
```
业务目标：提升用户留存
↓
指标候选：
- 次日留存率
- 7日留存率
- 30日留存率
- 月活跃用户数（MAU）
- 用户生命周期价值（LTV）

选择：7日留存率（平衡灵敏度与稳定性）
```

### Phase 2: 指标分层

```markdown
## 北极星指标（1个）
反映产品核心价值，全团队对齐
- 示例：GMV、DAU、留存率

## 一级指标（3-5个）
直接影响北极星指标
- 示例：转化率、客单价、购买频次

## 二级指标（过程指标）
监控具体环节
- 示例：页面加载时长、点击率、跳出率

## 健康指标（护栏指标）
防止为了优化主指标而伤害其他
- 示例：投诉率、退货率、获客成本
```

### Phase 3: 指标定义

**每个指标必须明确：**
```markdown
### 指标名：7日留存率

**计算公式**
7日留存率 = 第7天仍活跃的用户数 / 首日新增用户数 × 100%

**数据口径**
- "活跃"定义：打开APP并停留>10秒
- 时间窗口：自然日（0:00-24:00）
- 用户范围：新注册用户（注册当天为D0）

**目标值**
- 现状：15%
- 目标：25%
- 行业参考：30%（竞品平均水平）

**采集方式**
- 埋点事件：app_launch, app_exit
- 计算周期：T+1日报表
- 看板位置：BI系统-留存分析
```

### Phase 4: 指标陷阱检查

**避免以下问题：**
- [ ] 虚荣指标（总下载量，不看活跃）
- [ ] 后验指标（无法指导行动）
- [ ] 指标过多（看不过来）
- [ ] 目标值拍脑袋（无依据）

## Output

```markdown
# 指标体系设计

## 北极星指标
7日留存率（目标：25%）

## 指标看板

| 指标 | 定义 | 现状 | 目标 | 采集方式 |
|------|------|------|------|---------|
| 7日留存率 | D7活跃/D0新增 | 15% | 25% | 埋点计算 |
| 次日留存率 | D1活跃/D0新增 | 35% | 45% | 埋点计算 |
| 人均使用时长 | 总时长/DAU | 5min | 8min | 埋点计算 |
| 功能使用率 | 使用某功能/DAU | 20% | 35% | 埋点计算 |

## 健康指标（护栏）
- 投诉率 < 0.5%
- 页面崩溃率 < 0.1%

## 数据看板设计
[描述看板布局]
```
