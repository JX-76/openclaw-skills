# Prioritization - 需求优先级排序

---

## Description

在资源有限的情况下，确定需求/功能的优先开发顺序。

**解决的实际问题：**
- 需求太多，开发资源不够
- 各方争论先做什么
- 老板和业务方需求冲突

---

## When to Use

**强制触发条件（任一满足）：**
- 用户明确说"排优先级"、"排序"、"先做什么"
- 提供的需求列表 > 5 项
- 明确提到资源限制（时间/人力/预算）

**示例触发句：**
- "这10个需求排一下优先级"
- "开发说只能做3个，选哪几个？"
- "Q2资源有限，先上哪些功能？"

---

## Pre-conditions

**必须提供：**
- 待排序的需求列表（≥2项）
- 约束条件（时间/人力/预算，至少一项）

**可选提供：**
- 当前产品阶段（探索期/增长期/成熟期）
- 近期业务目标

---

## Workflow

### Phase 1: 评估维度定义

**根据产品阶段选择维度：**

| 产品阶段 | 核心维度 | 权重 |
|---------|---------|------|
| 探索期 | 用户价值、验证速度 | 50%/50% |
| 增长期 | 用户价值、商业价值、技术成本 | 40%/40%/20% |
| 成熟期 | 商业价值、技术成本、风险评估 | 40%/30%/30% |

### Phase 2: 逐项评分

**评分标准（1-5分）：**

```markdown
## 用户价值
5 = 解决核心痛点，不用产品就废
4 = 显著提升体验，竞品已有
3 = 体验优化，锦上添花
2 = 边缘需求，少数用户需要
1 = 几乎无价值

## 商业价值
5 = 直接带来收入/降本，ROI清晰
4 = 间接促进转化/留存
3 = 长期战略价值
2 = 可能有用但不确定
1 = 纯成本项

## 技术成本
5 = 1周内可完成
4 = 2周内可完成
3 = 1月内可完成
2 = 需要架构改动
1 = 技术风险高，周期不确定
```

### Phase 3: 计算优先级分数

```
优先级分数 = 用户价值 × 权重1 + 商业价值 × 权重2 + 技术成本 × 权重3

示例（增长期：用户40%/商业40%/成本20%）：
需求A：用户4 × 0.4 + 商业5 × 0.4 + 成本3 × 0.2 = 4.2
需求B：用户5 × 0.4 + 商业3 × 0.4 + 成本4 × 0.2 = 3.8
```

### Phase 4: 资源匹配

**根据资源约束调整：**
```markdown
## 资源约束：2个开发，1个月

### P0（必须做）
- 需求A（4.2分，2周）
- 需求C（3.9分，1.5周）

### P1（有余力就做）
- 需求B（3.8分，2周）← 剩余0.5周，延期风险

### P2（下期）
- 其他需求
```

---

## Output

```markdown
# 需求优先级排序结果

## 排序逻辑
- 产品阶段：增长期
- 评估维度：用户价值(40%) + 商业价值(40%) + 技术成本(20%)
- 资源约束：2开发 × 4周

## 优先级结果

### P0 - 本期必须做
| 需求 | 综合分 | 预估工期 | 理由 |
|------|--------|---------|------|
| 会员系统 | 4.5 | 2周 | 高用户价值+高收入潜力 |
| 优惠券 | 4.2 | 1.5周 | 直接促进转化 |

### P1 - 有余力就做
| 需求 | 综合分 | 预估工期 | 风险 |
|------|--------|---------|------|
| 积分商城 | 3.8 | 2周 | 工期紧，建议延期 |

### P2 - 下期规划
| 需求 | 综合分 | 建议时机 |
|------|--------|---------|
| 社区功能 | 3.2 | 用户量突破10万后 |

## 关键决策建议
1. 会员系统和优惠券先做，预留1周buffer
2. 积分商城如要本期做，需砍掉部分功能
3. 建议P2需求暂不投入，避免分散资源
```
